# Prompt Refinement Features â€” Design Spec

## Overview

Add post-generation refinement capabilities to the prompt engineer. After receiving their enhanced prompt, users can tweak parameters, apply pro tips, use free-text refinement instructions, and track a full version history of all changes. Static tips apply client-side instantly. Steering dial changes and free-text refinements trigger API calls on demand.

## Scope

- Refinement panel on Screen 3 (pro tips, steering dials, free-text refine input)
- AI-generated tips returned alongside synthesis response
- Static category best practices (client-side config)
- Free-text refinement via new API endpoint
- Steering dial re-generation (reuses existing synthesis endpoint)
- Version history with full stack navigation

## Out of Scope

- Collaborative editing / sharing refinements
- Prompt diffing UI (visual character-level diff between versions)
- A/B comparison of two versions side by side

---

## Refinement Panel

Sits below the changelog on Screen 3. Collapsible, expanded by default on first generation.

### Layout (top to bottom)

1. **Pro Tips Section** â€” Two subsections:
   - **AI Suggestions** â€” 3-5 specific, actionable tips generated by Claude alongside the final prompt. Each is a clickable chip. Clicking applies the tip client-side as a text insertion into the prompt. Applied tips get a checkmark and dim out.
   - **Category Best Practices** â€” 2-3 static tips based on the detected category (defined in a client-side config file keyed by category). Same clickable chip behavior as AI suggestions.

2. **Steering Dials** â€” The same universal + category-specific dials from Screen 1, pre-filled with the values the user originally set. Adjusting these queues a regeneration but does not trigger it immediately.

3. **Free-Text Refine Input** â€” A textarea with placeholder "Describe how you'd like to refine this prompt..." and a "Refine" button. Sends the instruction to Claude along with the current prompt to produce an updated version.

4. **"Refresh Prompt" Button** â€” Appears when steering dials have been changed from their current values. Batches all dial changes into one regeneration call.

### Behavior

- **Tip application (client-side, instant):** Clicking a tip chip appends the tip as an instruction/constraint to the current prompt text. A new version is created in the version stack. No API call.
- **Free-text refinement (API call):** User types instruction, clicks "Refine." Calls `POST /api/sessions/[id]/refine` with the current prompt and instruction. Claude returns an updated prompt and a short description of what changed. New version created.
- **Steering dial regeneration (API call):** User adjusts dials, clicks "Refresh Prompt." Calls the existing synthesis endpoint with the original raw prompt, original answers, updated steering values, and target model. Full regeneration. New version created labeled with what changed.

## Pro Tips

### AI-Generated Tips

Generated in the same API call as the synthesis. The synthesis prompt template is extended to instruct Claude to also return a `tips` array of 3-5 specific, actionable suggestion strings alongside `finalPrompt` and `changelog`.

Example tips:
- "Add an example of the desired output format to guide the AI's response structure"
- "Specify the target word count or length constraint"
- "Include a persona or role for the AI to adopt"
- "Add negative constraints â€” what the output should NOT include"

Tips are specific to the prompt content, not generic advice.

### Category Best Practices

Static tips defined client-side in a config file, keyed by `PromptCategory`. These are general best practices, not specific to the prompt content.


### Tip Application Behavior

When a user clicks a tip chip:
1. The tip text is appended to the current prompt as a new constraint or instruction line (e.g., `"\n\nAdditional constraint: [tip text]"`)
2. A new version entry is created: `{ trigger: "tip", description: "Applied: [tip text]", prompt: [updated prompt] }`
3. The tip chip shows a checkmark and dims out
4. The prompt display updates immediately â€” no API call
5. Tips are one-way â€” once applied, the user can undo by navigating to a previous version in the version history

## Version History

### Location

A compact version indicator appears next to the "Enhanced Prompt" heading in the prompt card header. Shows "v3 of 3" style text. Clicking expands a dropdown/popover listing all versions.

### Version Entry

Each version entry contains:
- `id`: unique string (cuid or incremental)
- `prompt`: the full prompt text at this version
- `changelog`: array of changelog strings (from synthesis, or generated description for refinements)
- `tips`: array of AI-generated tip strings (only present on synthesis versions)
- `trigger`: what created this version â€” `"synthesis"` | `"tip"` | `"refine"` | `"regenerate"`
- `description`: human-readable label (e.g., "Initial generation", "Applied: Add output format example", "Refined: made more concise", "Adjusted tone to formal")
- `createdAt`: ISO timestamp

### Behavior

- Every change that modifies the prompt creates a new version â€” tip applications, free-text refinements, and steering dial regenerations.
- Clicking any version in the list loads that version's prompt into the display card.
- The active version is highlighted in the list.
- Jumping to an old version does NOT delete newer ones â€” the user can navigate freely.
- If the user makes a new refinement from an old version, the new version is appended to the end of the stack. All previous versions remain accessible.
- The copy button always copies the currently displayed version.

### Storage

The version stack is stored on the `PromptSession` in a new `versions` JSON field. This persists across page reloads and is loaded when a session is selected from the sidebar.

## Service Layer Changes

### SynthesisService

- Update `safeJsonParse` type in `synthesize()` to expect `{ finalPrompt, changelog, tips }` instead of `{ finalPrompt, changelog }`
- Thread `tips` through to the `SynthesisResponse` object
- Persist `tips` on the session alongside `finalPrompt` and `changelog`
- When building the initial version entry, include `tips` from the synthesis response

### New: RefinementService

A new service in `packages/services/src/refinement.service.ts` that handles:
- Free-text refinement (calls Claude with refinement prompt)
- Steering dial regeneration (calls Claude with synthesis prompt + new steering values, reading original rawPrompt/answers/targetModel from session)
- Version persistence (appends to the `versions` JSON array on the session)

### SessionService / loadSession Hook

- `loadSession` in `use-session-flow.ts` must restore `steeringInputs` and `targetModel` from the session data so the dials on Screen 3 show the user's original values (not defaults)
- The `SessionResponse` validator must be extended to include `versions`, `steeringInputs`, and `targetModel` fields

## New API Endpoints

### `POST /api/sessions/[id]/refine`

**Request:** `RefineInput` â€” `{ currentPrompt, instruction }`

**Behavior:**
1. Load the session, verify it's COMPLETED
2. Call Claude with a refinement prompt template
3. Return `RefineResponse` â€” `{ finalPrompt, description }`
4. Update the session's `versions` array with the new version
5. Update `finalPrompt` on the session to the latest version

**Does NOT change session status** â€” stays COMPLETED throughout refinements.

**No tier gating** â€” refinements are available to all users during testing (same as analysis/synthesis).

### `POST /api/sessions/[id]/regenerate`

**Request:** `RegenerateInput` â€” `{ steeringInputs }`

**Behavior:**
1. Load the session, verify it's COMPLETED
2. Read original `rawPrompt`, `answers`, `targetModel` from session
3. Update session's `steeringInputs` with the new values
4. Call synthesis with the updated steering context
5. Return `SynthesisResponse` (with new tips)
6. Append new version to `versions` array
7. Update `finalPrompt` on the session

New schema:

```typescript
export const RegenerateInput = z.object({
  steeringInputs: SteeringInputs,
});
export type RegenerateInput = z.infer<typeof RegenerateInput>;
```

## New Prompt Template

### Refinement Prompt (`packages/prompts/src/refinement.prompt.ts`)

```
You are an expert prompt engineer. You are refining an existing optimized prompt based on a user's instruction.

Given the current prompt and the user's refinement instruction, produce an updated version of the prompt.

Rules:
- Apply the refinement precisely â€” don't change unrelated parts
- Maintain the overall structure and quality of the prompt
- If the instruction is vague, interpret it reasonably
- Keep the prompt self-contained

Return JSON with two fields:
- "finalPrompt": string (the updated prompt)
- "description": string (1 sentence describing what you changed)

Return ONLY valid JSON. No markdown, no explanation.
```

The target model is read from the session and passed to this template (same model-specific formatting guidance as synthesis).

```
Return JSON with three fields:
- "finalPrompt": string (the complete optimized prompt)
- "changelog": string[] (3-5 improvement descriptions)
```
